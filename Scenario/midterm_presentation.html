프로젝트 중간 발표 (Midterm)

0. 기존 프로젝트 대비 주요 변경점

플랫폼의 확장 및 로컬 AI 파이프라인 도입

  • 1. 구동 플랫폼 진화 (데스크톱 프로그램 ➔ 웹 앱)
    기존의 데스크톱 종속 프로그램에서 벗어나, React 기반의 웹 애플리케이션(Web App)으로 전환하였습니다.
  • 2. AI 아키텍처 전환 (클라우드 ➔ 로컬 AI 하이브리드)
    단순 외부 API(OpenAI 등) 의존 방식에서 특화된 로컬 모델 (Gemma 등)을 자체 서버에서 구동하는 방식으로 전환하여 보안과 토큰 비용 문제를 해결했습니다. (필요시 온라인 모델 병행 사용 가능)
  • 3. 데이터 자동화 (AI 자동 문맥 분석 및 텍스트 추출)
    사용자가 일일이 수동으로 항목을 채우던 기존과 달리, 소설 등 텍스트 원문만 입력하면 AI가 스스로 고유 특성을 인지하여 인물/장소/플롯 데이터를 자동 추출하고 DB에 적재하는 파이프라인이 추가되었습니다.

1. 현재 진행 상황

로컬 AI 및 RAG 파이프라인 연동 현황

  • 엔티티 추출 파이프라인: 유저 텍스트 입력 시, Ollama 로컬 모델을 통해 인물/장소/플롯 파편 데이터를 매핑하고 JSON 형식으로 추출하도록 구현 완료됨.
  • 무결성 검사 연동: Microsoft GraphRAG 기술을 내장하여, 기존 설정과 새로 입력된 데이터 간의 논리적 충돌 가능성을 검증하는 프로세스 연동.
  • 데이터 적재 완료: 검증된 데이터를 Web App 내 세계관 메타데이터 테이블(Worldview DB)에 구조적으로 저장함.
🔗 참조: MS GraphRAG Repository

2. 최종 목표 (시각화 UI 구현)

단순 요약 텍스트를 넘어선 계층적 지식 탐색 경험

  • 시각화 트리거: 개체 데이터(예: '홍길동' 프로필) 상단의 '요약' 버튼 우측에 [시각화 버튼] 신설 타겟.
  • 동적 마인드맵 렌더링: 버튼 클릭 시 해당 개체를 중심 노드로 하는 RAG 연관 지식 그래프(Mind Map) 동적 표출.
  • 복합 필터링 팝업 UI: 화면 우측 팝업을 통해 [인물 / 플롯 / 장소] 별로 노드 필터링 제어.

    ※ 특정 플롯 필터를 적용할 경우, 연관된 장소 노드가 일렬 나열되지 않고 플롯 노드에서 파생된 가지(Branch) 형태로 시각적인 계층 연결망 생성

🔗 활용 예정 기술: React Flow DB Visualization

2-1. 최종 목표 (최적화 및 확장성 검토)

데이터 파이프라인의 핵심 성능 및 부가 기능 조율

  • ① 현재 추출 파이프라인 수치 개선 (최우선 과제)
    현재 모델이 가진 데이터 무결성 한계를 돌파하기 위해 실험과 파인튜닝을 통해 AI의 오답을 통제하는 정밀도(Precision), 그리고 필수 정보의 누락을 방어하는 재현율(Recall) 지표를 극대화하는 것에 총력을 기울입니다.
  • ② AI 이미지 생성 기능 (기능적 가능성 검토)
    구축된 텍스트 에셋을 시각적 이미지로 변환해주는 AI 일러스트 생성 자동화 기능은 본 프로젝트의 메인 요소가 아니므로, 향후 개발의 확장 가능성(Possibility) 수준에서만 남겨두기로 우선순위를 책정합니다.

3. 차후 계획 및 실험

SFT 파인튜닝을 적용한 '로컬 AI 텍스트 추출 정확도' 고도화

  • 문제점: 현 Zero-shot 프롬프트 추출(JSON) 환경에서 포맷 무결성 훼손 및 복잡한 관계 추출 누락 빈번.
  • 파인튜닝(SFT) 플랜: Baseline 실험에 사용된 Gemma 계열 로컬 LLM을 개체 식별(NER)에 맞게 지도 학습(LoRA).
    🔗 사전학습 모델: Google Gemma 모델 (HuggingFace)
  • 학습 데이터셋 투입 계획: 판타지, 소설 세계관 특화 NER(개체명 인식) 데이터셋 활용 플랜.
    🔗 활용 데이터셋: HF Fiction NER (약 7.5억 토큰 픽션 데이터)
  • 실측 결과 및 고도화 목표 (Target Metrics, 200샘플 기준):
    • JSON 포맷 에러율: 현재 gemma4:26b 16.5% ➔ 1% 미만 달성 목표 (SFT 후)
    • 추출 모델 F1-Score: 현재(Zero-shot) 0.409 ➔ 0.90 이상 타겟 향상

3-1. 2번째 학습안 (데이터 구축 파이프라인)

고품질의 자체 SFT 데이터셋 (목표: 2,000개) 구축

  • a. 단편소설 데이터 확보
    Kaggle 등 오픈 데이터 저장소를 활용하여 적절한 길이의 단편소설 데이터셋을 수집합니다.
  • b. 상용 LLM 기반 정답(JSON) 추출
    수집된 단편선들을 Claude 등 최신 상용 LLM에 입력하여 정교한 구조의 JSON 포맷(Key-Value) 데이터로 자동 추출합니다.
  • c. 모범답안(Gold Standard) 데이터셋 병합
    프롬프트로 사용된 단편선 원문(Story)과 추출된 JSON 결과물을 한 쌍으로 묶어, 우리 모델을 파인튜닝할 최종 모범답안 데이터셋을 완성합니다.

4. 평가 지표 (Evaluation Metrics)

정보 추출(IE) 성능 평가를 위한 계산 수식 및 채택 사유

  • 정밀도 (Precision) : 0.529
    (올바르게 맞춘 추출 개체 수) ÷ (모델이 추출해낸 전체 개체 수) ➔ AI 환각(할루시네이션) 방어력 지표
  • 재현율 (Recall) : 0.334
    (올바르게 맞춘 추출 개체 수) ÷ (실제 원문 내 정답 개체 총 수) ➔ 추출 누락 검증 (촘촘함) 지표
  • F1-Score : 0.409
    2 × (Precision × Recall) ÷ (Precision + Recall) ➔ 두 지표의 조화 평균(Harmonic Mean)
※ 지표 채택 사유 (Why F1-Score?):

단순히 맞춘 비율을 보는 단순 정확도(Accuracy)는 빈 데이터(일반 단어)가 많은 '개체 추출(NER)' 환경에서 성능을 90% 이상으로 착각하게 만드는 심각한 왜곡을 낳습니다. 따라서 '오답을 뱉지 않는 능력(Precision)'과 '정답을 빠뜨리지 않는 능력(Recall)'의 균형을 보는 조화평균(F1-Score)을 사용하는 것이 학계의 엄격한 표준 평가론입니다.
