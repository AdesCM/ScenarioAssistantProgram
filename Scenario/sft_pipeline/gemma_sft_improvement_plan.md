# Gemma SFT 모델 성능 개선 및 하이퍼파라미터 최적화 방안

본 문서는 소설 시나리오 개체 추출 태스크에서 Gemma SFT(Supervised Fine-Tuning) 모델의 F1-Score를 **0.8 목표**로 끌어올리기 위한 학습 구조 및 하이퍼파라미터 개선 계획을 정리한 문서입니다. (발표용 PPT 자료 활용 목적)

---

## 1. 하이퍼파라미터 변경 대비표 (현재 vs 개선안)

| 하이퍼파라미터 | 현재 SFT 설정 | **개선 최적화 설정 (Proposed)** | 개선 목적 및 기대 효과 |
| :--- | :---: | :---: | :--- |
| **베이스 모델 (Model ID)** | `google/gemma-4-E4B-it` (가칭) | **`google/gemma-2-2b-it`** 또는 **`9b-it`** | 공식 지원되는 최신 Gemma 2 모델을 활용하여 기초 언어 독해력 극대화 |
| **학습률 (Learning Rate)** | `2e-4` (0.0002) | **`5e-5` ~ `1e-4`** | 과도한 가중치 변화를 억제하여 모델의 텍스트 매핑 주의집중(Attention) 안정화 |
| **실질 배치 크기 (Effective BS)** | `8` (1bs × 8accum) | **`16` ~ `32`** (1bs × 16/32accum) | 그래디언트 노이즈를 제어하고 국소 최적해(Local Minima) 탈출을 유도 |
| **LoRA Rank (r)** | `32` | **`16`** | 어댑터 크기를 줄여 모델이 스키마(JSON 뼈대)만 암기하는 오버피팅 방지 |
| **LoRA Alpha ($\alpha$)** | `64` | **`32`** | 랭크 하향에 맞춘 스케일 최적화 ($r \times 2$) |
| **LoRA Dropout** | `0.05` | **`0.1`** | 학습 규제(Regularization)를 강화하여 스키마 구조 단순 암기 억제 |
| **최대 시퀀스 길이 (Max Length)** | `512` | **`1024`** | 긴 소설 텍스트 본문과 상세한 JSON 출력을 누락 없이 수용 |
| **프롬프트 템플릿 (Prompt Template)** | Raw 텍스트 형식 | **Gemma 공식 Chat Template** | Gemma 특수 토큰 구조에 맞춰 지시어 이행(Instruction Following) 극대화 |
| **데이터 전처리 및 토큰화** | 경계 구분 개별 인코딩 | **통합 인코딩 & 슬라이싱 마스킹** | Prompt-Target 경계 지점의 토큰 매핑 불일치(Boundary Mismatch) 원천 차단 |

## 2. 학습 구조 개선 요약 (Key Improvements)

### ① Prompt-Target 통합 인코딩 및 손실 마스킹(Loss Masking) 정밀화
* **현상**: Prompt와 Target을 각각 인코딩하여 결합하는 기존 방식은 인코딩 경계면에서 토큰이 쪼개지는 파편화 문제(Token Mismatch)를 유발하여 추론 성능 붕괴의 원인이 됨.
* **개선**: 전체 텍스트(`Prompt + Target`)를 통째로 토크나이징한 뒤, Prompt 영역 토큰 개수를 계산해 해당하는 레이블만 `-100`으로 마스킹하는 방식으로 전환하여 추론 환경과의 일치도 100% 확보.

### ② Gemma 전용 공식 Chat Template 도입
* **현상**: Gemma 모델은 특수 메타 토큰(`<start_of_turn>user\n...`) 포맷에 강하게 종속되어 있어 이를 따르지 않을 경우 지시 이해도가 감소함.
* **개선**: 학습 및 평가 시 입력 프롬프트를 Gemma 공식 대화 템플릿 양식으로 포장하여 지시 이행 및 구조화 JSON 출력 완결성을 확보.

### ③ 시퀀스 길이 확장 (512 $\rightarrow$ 1024)
* **현상**: 소설 본문의 상세한 배경 묘사나 성격 묘사 정보를 추출할 때, 시퀀스 길이가 512로 제한되어 있어 후반부 정답 데이터가 잘려 나감.
* **개선**: 최대 길이를 1024 토큰으로 늘려 복잡한 디테일 정보를 누락 없이 완벽히 추출할 수 있도록 보장.

## 3. 파라미터 개선 요약 (Hyperparameter Optimizations)

### ① 표현 붕괴(Representation Collapse) 현상 방지
* **원인**: 학습 초기 너무 높은 학습률(`2e-4`)과 큰 LoRA Rank(`32`)로 인해, 본문의 텍스트 내용을 세부정보에 채워 넣는 복잡한 주의력 학습보다 빈 문자열 JSON 틀만 채우는 단순 패턴 학습(Shortcut)에 먼저 수렴해 버리는 부작용이 발생함.
* **조치**: LoRA Rank를 `16`으로 타협하고 학습률을 `5e-5`로 하향 조정함으로써 모델 고유의 추출 및 독해 능력을 보존하며 타겟 포맷만을 학습하도록 유도.

### ② 배치 노이즈 억제 및 안정성 강화
* **조치**: Gradient Accumulation 단계를 `8`에서 `16`~`32`로 늘려 실질 배치 크기(Effective Batch Size)를 늘리고 코사인 감쇠 스케줄러(Cosine Scheduler)와 웜업 비율(Warmup Ratio)을 5% ~ 10% 수준으로 조율하여 부드러운 수렴을 달성.

### ③ 에포크(Epoch) 설정과 과적합(Overfitting) 경계
* **원인**: 모델 성능이 안 나온다고 에포크를 무작정 높이는 것(예: 10 ~ 50 에포크)은 LLM 파인튜닝에서 심각한 **과적합(Overfitting)과 환각(Hallucination)**을 유발합니다. 학습 데이터에 있는 인물 이름이나 사건을 그대로 외워버려서, 새로운 소설이 들어와도 학습했던 소설의 인물을 출력하는 치명적 오류가 발생할 수 있습니다.
* **조치**: SFT(지시어 튜닝)에서는 통상적으로 **3 ~ 5 에포크**가 최적의 스위트 스팟(Sweet Spot)입니다. 만약 5 에포크 내외에서도 포맷이 무너지거나 내용 추출이 안 된다면, 에포크를 늘릴 것이 아니라 앞서 언급한 **프롬프트 템플릿 정렬**과 **토큰 마스킹 정밀화**를 고치는 것이 기술적으로 올바른 해결책입니다.
